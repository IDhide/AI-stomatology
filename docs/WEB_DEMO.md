# 🌐 Веб-демо Smile.AI — визуал, камера и симуляция DIKIDI

Этот демо-стенд позволяет посмотреть **как выглядит интерфейс** ассистента и как
система реагирует на появление человека — **без GPU, без Ollama, без физического
дисплея**. Всё открывается в обычном браузере.

Боевой UI (`src/ui/display.py`) работает на pygame и требует подключённого
экрана. Веб-версия повторяет тот же визуал (медузы → пульсирующий круг с
аудио-волной → субтитры) и удобна для разработки и демонстрации.

---

## 0. Запуск через Docker (рекомендуется для теста)

Лёгкий стенд из двух контейнеров, только pure-python зависимости — без GPU,
Ollama, камеры и pygame. Два сервиса общаются по внутренней сети Docker:

```
web (8080)  ──HTTP──►  dikidi-sim (8089)
   киоск/визуал          симулятор DIKIDI API
```

```bash
# собрать и запустить
docker compose -f docker-compose.demo.yml up --build
#   или короче:
make demo

# открыть в браузере (ВАЖНО: Chrome или Edge — нужен Web Speech API)
#   http://localhost:8080            — интерактив: Оливия говорит и слушает микрофон
#   http://localhost:8080/?camera=1  — + распознавание людей через веб-камеру
#   http://localhost:8089/healthz    — проверка симулятора DIKIDI

# остановить
docker compose -f docker-compose.demo.yml down     # или: make demo-down
```

### 🔊 Локальный голос (Piper TTS) — рекомендуется для медицины
По умолчанию Оливия говорит через `speechSynthesis` браузера: голос зависит
от ОС/браузера и в Firefox для русского часто роботичный. Серверный **Piper**:
- одинаковый русский голос (`ru_RU/dmitri/medium`, ~63 МБ) в любом браузере;
- работает локально — звук пациента и голос Оливии **никуда не уходят**;
- быстро на CPU (≈0.5–1.5 с на короткую реплику).

Включается автоматически в `make smart` (билд-аргумент `WITH_TTS=1`).
Браузер сначала спрашивает `/api/tts/status` и переключается на серверный TTS,
если он есть; иначе фолбэк на голос браузера.

При первом запросе модель голоса скачивается в volume `piper_voices` и кэшируется.

### 🧠 Умный режим (Ollama LLM) — рекомендуется
По умолчанию «мозг» — простые правила (`responder.py`): отвечает по ключевым
словам, не понимает контекст. Чтобы Оливия понимала свободную речь и держала
нить диалога, подключите настоящую языковую модель (Ollama):

```bash
# демо + Ollama (поднимет модель и подключит её к киоску)
docker compose -f docker-compose.demo.yml -f docker-compose.smart.yml up --build
#   короче:
make smart
#   для Firefox с распознаванием речи:
WITH_STT=1 make smart
#   модель помощнее (нужен GPU/много RAM):
OLLAMA_MODEL=qwen2.5:7b-instruct make smart
```

- Первый запуск **скачивает модель** (~1.9 ГБ для `qwen2.5:3b`) — подождите,
  пока контейнер `smile-ollama-pull` завершится.
- Персона берётся из `config/prompts.yaml` (Оливия, цены, акции, правила) —
  модель отвечает уже «в роли», с учётом контекста и DIKIDI-инструментов.
- Если Ollama недоступен — киоск автоматически откатывается на правиловый
  `responder`, голос и визуал продолжают работать.
- Без Docker: запустите Ollama (`ollama serve`), `ollama pull qwen2.5:3b`,
  затем `OLLAMA_HOST=http://localhost:11434 OLLAMA_MODEL=qwen2.5:3b
  python3 -m src.web.server …`.

### Интерактивный голосовой режим (по умолчанию)
Откройте страницу, **коснитесь экрана** (браузер требует жест для микрофона и
звука) и **разрешите доступ к микрофону**. Дальше:

```
Оливия здоровается голосом → СЛУШАЕТ вас (STT) → ДУМАЕТ → ОТВЕЧАЕТ голосом (TTS)
```

- **Голос (TTS)** — `speechSynthesis` браузера, русский голос.
- **Слух (STT)** — `SpeechRecognition` (работает в Chrome/Edge).
- **Мозг** — `src/web/responder.py`: отвечает по персоне «Стоматология №1»
  (цены под ключ, акции, бесплатная консультация, удаление → партнёр-хирург,
  не лечим детей, запись с согласованием времени администратором). Ollama не нужен.
- `Esc` — остановить сессию.

**Распознавание речи — три пути (выбирается автоматически):**

| Браузер            | Как слышит пациента                                  |
|--------------------|-----------------------------------------------------|
| Chrome / Edge      | Web Speech API прямо в браузере (ничего не нужно)   |
| Firefox и др.      | MediaRecorder → `POST /api/stt` → faster-whisper    |
| STT недоступен     | строка ввода текстом (Оливия отвечает голосом)      |

### Firefox: распознавание на сервере
Firefox не умеет Web Speech API, поэтому речь распознаётся на сервере через
**faster-whisper** (русская модель, без torch). Нужно её включить:

```bash
# локально
pip install -r requirements-voice.txt
DIKIDI_BASE_URL=http://127.0.0.1:8089 python3 -m src.web.server --host 0.0.0.0 --port 8080

# Docker
WITH_STT=1 docker compose -f docker-compose.demo.yml up --build
#   размер модели:  STT_MODEL=base WITH_STT=1 docker compose ... up --build
```

- Модель скачивается с HuggingFace при первом запросе (≈ 0.5 ГБ для `small`,
  ≈ 150 МБ для `base`) и кэшируется. Нужен доступ в интернет к HuggingFace.
- Переменные: `STT_MODEL` (tiny|base|small|medium), `STT_DEVICE` (cpu|cuda),
  `STT_COMPUTE` (int8|float16|…).
- Если модель не установлена/недоступна — `/api/stt` вернёт 503, и страница
  автоматически покажет строку ввода текстом (голос Оливии при этом работает).

> В Firefox микрофон работает на `http://localhost` или по HTTPS.

### Демо-режим (заготовленный диалог, без микрофона)
Чтобы просто посмотреть визуал без разговора — запустите с флагом `--demo`:

```bash
python3 -m src.web.server --host 0.0.0.0 --port 8080 --demo
```

Логи диалогов пишутся в `./data/logs/` (примонтировано наружу). Чтобы
показать настоящее видео медуз — положите файл в `./assets/videos/jellyfish.mp4`.

> Полный боевой стек (Ollama + torch + pygame) — в `docker-compose.yml`.
> Здесь же — именно лёгкая версия для быстрой проверки визуала и логики.

## 2. Запуск веб-киоска (визуал + сценарий)

```bash
python3 -m src.web.server --port 8080
```

Откройте <http://127.0.0.1:8080> — пойдёт авто-сценарий:

```
idle (медузы) → 👤 «лицо обнаружено» → greeting → диалог о кариесе
            → запрос свободных окон у DIKIDI → запись → 👋 уход → idle
```

Режимы и их визуал:

| Режим      | Что видно                                                        |
|------------|-----------------------------------------------------------------|
| `idle`     | Медузы / звёздный фон (ожидание)                                 |
| `greeting` | Круг проявляется, лёгкая «волна»                                 |
| `listening`| Зелёный круг, спокойное «дыхание» волны                          |
| `thinking` | Янтарный круг, мелкая рябь                                       |
| `speaking` | Фиолетовый круг + аудио-волна в такт речи + субтитры             |

Флаги:

```bash
python3 -m src.web.server --port 8080 --no-loop   # без авто-сценария
curl -X POST http://127.0.0.1:8080/api/trigger    # вручную «впустить пациента»
```

Горячие клавиши на странице (для отладки без сервера): `1`–`5` переключают режимы.

## 3. Камера и распознавание людей

«Железный» путь MVP — обычная USB веб-камера → браузер. Включается флагом:

```
http://127.0.0.1:8080/?camera=1
```

`web/camera.js`:
- запрашивает доступ к камере (`getUserMedia`),
- детектирует движение (покадровая разница) и лицо (`FaceDetector API`, если есть),
- при устойчивом обнаружении человека шлёт `POST /api/trigger` → запускается
  приветствие и диалог,
- в углу показывает self-view и статус («камера активна» / «человек обнаружен»).

> В headless-окружении (CI/контейнер без камеры) используйте авто-сценарий или
> ручной триггер — логика присутствия там симулируется на сервере.

Боевая детекция на Python (MediaPipe + OpenCV, RTSP/USB) — в
`src/camera/detector.py`.

## 4. Симуляция DIKIDI API

Есть два уровня подмены DIKIDI:

### a) In-process stub (`src/dikidi/client_stub.py`)
Подменяет клиент прямо в памяти — используется веб-сервером и `main_offline.py`.

### b) HTTP-симулятор (`src/dikidi/fake_server.py`) — «настоящий» API
Поднимает HTTP-сервер с теми же путями, что у боевого DIKIDI, поэтому
**продакшн-клиент `src/dikidi/client.py` работает с ним без изменений**.

```bash
# запустить симулятор API
python3 -m src.dikidi.fake_server --port 8089 --no-auth

# проверить
curl http://127.0.0.1:8089/healthz
curl "http://127.0.0.1:8089/v1/company/services"
```

Эндпоинты:

| Метод | Путь                            | Назначение            |
|-------|---------------------------------|-----------------------|
| GET   | `/healthz`                      | проверка живости      |
| GET   | `/v1/company/services`          | список услуг с ценами |
| GET   | `/v1/company/masters`           | список врачей         |
| GET   | `/v1/booking/available-slots`   | свободные окна        |
| POST  | `/v1/booking/create`            | создать запись        |
| GET   | `/v1/clients/search`            | поиск клиента         |

Чтобы боевой клиент ходил в симулятор, укажите в `.env` / `settings.yaml`:

```env
DIKIDI base_url = http://127.0.0.1:8089
```
